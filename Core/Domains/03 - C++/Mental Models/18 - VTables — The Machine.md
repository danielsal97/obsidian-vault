# VTables — The Machine

## The Model

A vtable is an array of function pointers, one per virtual function, generated by the compiler at compile time and stored in the binary's `.rodata` section. Every class with virtual functions gets one vtable. Every instance of that class gets a hidden `vptr` (virtual pointer) as its first 8 bytes, pointing to the class's vtable. A virtual call is: load vptr → index into vtable → indirect call through function pointer.

---

## Memory Layout

```
class Animal {
    virtual void speak();    // vtable[0]
    virtual void move();     // vtable[1]
    int age;
};

class Dog : public Animal {
    void speak() override;   // replaces vtable[0]
    // move() not overridden: vtable[1] stays as Animal::move
    int breed;
};
```

```
Animal vtable (.rodata):
┌──────────────────┐
│ &Animal::speak   │  [0]
│ &Animal::move    │  [1]
└──────────────────┘

Dog vtable (.rodata):
┌──────────────────┐
│ &Dog::speak      │  [0]  ← overridden
│ &Animal::move    │  [1]  ← inherited
└──────────────────┘

Animal instance in memory:
┌──────────────┐
│ vptr         │ → points to Animal vtable
│ age          │
└──────────────┘

Dog instance in memory:
┌──────────────┐
│ vptr         │ → points to Dog vtable
│ age          │  (inherited from Animal)
│ breed        │  (Dog's own field)
└──────────────┘
```

## How It Moves — Virtual Dispatch

```cpp
Animal* a = new Dog();
a->speak();
```

```
At runtime:
      │
      ▼
a is a pointer to a Dog instance
  → a[0] = vptr pointing to Dog's vtable
      │
      ▼
Compiler generated call sequence:
  mov rax, [a]        ; load vptr from first 8 bytes of object
  mov rax, [rax + 0]  ; load vtable[0] = &Dog::speak
  call rax            ; indirect call through function pointer
      │
      ▼
Dog::speak() executes
  → 'this' pointer = a (adjusted if needed for multiple inheritance)
```

Compiled equivalent: `(a->vptr[0])(a)` — load pointer, dereference, call.

## Constructor Sets vptr

```
Dog d;

Execution order:
  1. Allocate storage for Dog (stack or heap)
  2. Animal's constructor runs:
       → sets d.vptr = &Animal::vtable   ← Animal's vtable!
       → initializes Animal's data members
  3. Dog's constructor runs:
       → sets d.vptr = &Dog::vtable      ← overwrites with Dog's vtable
       → initializes Dog's data members

After construction: d.vptr → Dog's vtable ✓
```

This is why calling `virtual` functions in a base class constructor is dangerous: the `vptr` hasn't been updated to the derived class yet. The base constructor sees the base's vtable, not the derived class's.

---

## Object Slicing — The Vtable Trap

```cpp
Dog dog;
Animal a = dog;    // copy Animal's DATA from dog, but NO vptr copy
```

```
After copy:
  a.vptr → Animal::vtable   (Animal's default vptr, not Dog's)
  a.age  = dog.age          (data copied)
  // dog.breed is GONE — Animal has no breed field
```

`a.speak()` calls `Animal::speak()`, not `Dog::speak()`. The Dog part is sliced off. Always use pointers or references for polymorphism.

---

## Multiple Inheritance — Multiple Vtables

```cpp
class Flyable { virtual void fly(); };
class Swimmable { virtual void swim(); };
class Duck : public Flyable, public Swimmable { ... };
```

Duck has TWO vtables (one for each base class interface). Duck instances have TWO vptrs. When casting a `Duck*` to `Swimmable*`, the pointer value is adjusted by the offset to the Swimmable subobject — the pointer changes address.

This is why comparing pointers from different inheritance branches requires care.

---

## Cost of Virtual Dispatch

Direct call: `a.speak()` where type is known → 1 instruction (direct `call` to address).

Virtual call: `a->speak()` where type is unknown → 3 extra instructions:
1. Load vptr from object
2. Index into vtable
3. Indirect call through pointer

Additional cost: indirect branch is harder to predict. Modern CPUs use indirect branch predictors (branch target buffers) but a call site with many possible targets (polymorphic hotspot) may mispredict ~5% of the time → 10-20 cycle penalty per mispredict.

Devirtualization: if the compiler can prove the concrete type at compile time, it eliminates the indirect call entirely. `final` keyword enables this.

---

## RTTI — Behind dynamic_cast

Every class with virtual functions has a `typeinfo` object (stored near the vtable in `.rodata`). `dynamic_cast<Dog*>(ptr)` uses RTTI to walk the inheritance hierarchy at runtime and verify the cast is valid.

Cost: walks the base-class chain. O(depth) with constant per level. For deep hierarchies: 10-100ns. Avoid in hot paths.
