# Assembler — The Machine

## The Model
A mechanical punch-card converter. One assembly instruction enters, one fixed-width binary pattern exits. No thinking, no understanding — pure lookup table translation. The blank sticky notes from the compiler survive unchanged.

## How It Moves

```
.s file (assembly text)
      |
      v
[LINE BY LINE]
  "mov rax, 5"    →  0x48 0xC7 0xC0 0x05 0x00 0x00 0x00
  "call [blank]"  →  0xE8 0x00 0x00 0x00 0x00  ← blank slot preserved
  "ret"           →  0xC3
      |
      v
.o file (ELF binary)
  — code section (.text): the binary instructions
  — data section (.data / .bss): global/static variables
  — relocation table: "byte offset 47 needs the address of LocalStorage::Read"
  — symbol table: "this .o exports the symbol InputMediator::Notify"
```

**WHY it's dumb by design:** The assembler's job is pure, parallel, and fast. Any intelligence added here would slow it down and duplicate work already done by the compiler.

## The Blueprint

- **ELF format**: the `.o` file is an ELF (Executable and Linkable Format) container with named sections
- `.text` section: executable instructions
- `.data` section: initialized global variables
- `.bss` section: zero-initialized globals (takes no space in file — just a size declaration)
- `.rodata` section: string literals, const data
- **Relocation entry**: `{offset: 47, type: R_X86_64_PLT32, symbol: "LocalStorage::Read", addend: -4}` — exact instructions for the linker to patch this slot

## Where It Breaks

Rarely. The assembler's input was generated by the compiler, which already validated the syntax. The only failure you see in practice:
- Assembler version mismatch (using an instruction the CPU doesn't support)
- Inline assembly you wrote by hand with a typo

## In LDS

`Makefile` — the `-c` flag in rules like:
```makefile
$(CXX) $(FLAGS) -c $< -o $@
```
The `-c` flag tells the compiler to invoke the assembler and stop. The `.o` files collected in `OBJ_CPP` are the assembler's output — binary blobs with relocation tables waiting for the linker.

Run `objdump -t services/mediator/InputMediator.o` on any LDS `.o` file to see its symbol table: which symbols it exports and which it needs filled in.

## Validate

1. The `.o` file is binary. The `.s` file is text. Can you reconstruct the `.s` from the `.o`? What information is lost?
2. Two `.cpp` files both define a `helper()` function (static, so no conflict). How does the assembler's output handle this — what prevents the linker from getting confused?
3. `objdump -d` on an LDS `.o` shows `callq  0x0`. What is that `0x0`, and who will fix it?
