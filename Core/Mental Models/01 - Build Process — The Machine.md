# Build Process — The Machine

## The Model
An assembly line with 4 stations, each enclosed in a sealed booth. Workers inside each booth cannot see into the next booth. Raw text enters station 1; a single executable binary exits station 4.

## How It Moves

```
 .cpp file (text)
      |
      v
[STATION 1: Preprocessor]  — text robot, no C++ knowledge
      | .cpp (expanded text, no directives)
      v
[STATION 2: Compiler]      — only station that understands C++
      | .s (assembly text, blank slots for external symbols)
      v
[STATION 3: Assembler]     — mechanical translator
      | .o (binary, blank slots still present as relocation entries)
      v
[STATION 4: Linker]        — only station that sees ALL .o files
      | executable (blank slots filled, addresses resolved)
      v
    ./LDS
```

WHY stations are sealed:
- Each station was designed and optimized for one job. Coupling them would mean changing the C++ grammar requires rewriting the linker.
- The compiler does not know how many .o files exist. The linker does not know C++. Isolation is not convenience — it is the only way the pipeline scales to millions of .cpp files across separate machines.

WHY blank slots exist:
- When the compiler sees `storage.Read(request)`, it emits a CALL instruction but cannot know the address of `LocalStorage::Read`. That address does not exist yet — it lives in a different .cpp file being compiled simultaneously on a different CPU core. The blank slot (relocation entry) is a promise: "fill this in later."

## The Blueprint

- **Preprocessor**: purely textual. Expands `#include`, substitutes `#define`, erases `#ifdef` branches. Output is one giant .cpp string. No tokens, no AST.
- **Compiler**: parses tokens → builds AST → semantic analysis → IR (GIMPLE/LLVM IR) → optimization passes → code generation → `.s` assembly text. Emits relocation entries for every unknown symbol.
- **Assembler**: reads `.s` line by line. Each mnemonic maps to a fixed binary opcode. Produces `.o` (ELF object file). Relocation table carried through unchanged.
- **Linker**: reads all `.o` files, builds a symbol table, walks every relocation entry, patches the binary with real addresses. Two modes:
  - **Static** (`-l:libfoo.a`): copies the needed `.o` slices from the archive directly into the output binary. Binary is self-contained.
  - **Dynamic** (`-lfoo` + `-Wl,-rpath`): writes a stub and a path. The dynamic linker (`ld.so`) fills the address at process launch.

In LDS the Makefile uses dynamic linking:
```makefile
# Makefile line 127
$(CXX) $(FLAGS) $(DFLAGS) -L$(LIB_DIR) -Wl,-rpath,$(LIB_DIR) $< -o $@ -l$(LIBRARY_NAME)-debug -pthread
```
The `-Wl,-rpath,$(LIB_DIR)` bakes the search path for `libfoo-debug.so` into the binary so it is found at runtime without `LD_LIBRARY_PATH`.

## Where It Breaks

- **Preprocessor**: wrong include path → file not found → station 1 halts. The compiler never sees the error; it never ran.
- **Compiler**: type mismatch, undeclared variable, syntax error → station 2 halts with the .cpp line number. `.o` file never produced.
- **Assembler**: rarely fails in practice; its input was generated by the compiler.
- **Linker**: `undefined reference to 'hrd41::LocalStorage::Read'` → a blank slot was never filled. Either the `.o` file is missing from the command, or the library was not linked. This is the most confusing error because the line number points nowhere — the C++ source compiled fine.
